Coordination patterns with OctoStore

OctoStore is a lock and lease primitive. With a few conventions, it becomes a simple coordination pattern for workers, agents, services, scripts, and CI jobs.

The pattern

  1. choose one stable job name
  2. try to claim it
  3. attach metadata about the owner
  4. renew while active
  5. release on completion
  6. let TTL expiry recover from crashes

Stable names matter

If the same logical work item is visible to multiple workers, they should all try to claim the same name.

  • images/thumbnail/img-123
  • deploy/prod
  • github/issues/123
  • cron/daily-summary/2026-04-01

Do not add random suffixes for the same logical task.

Metadata turns locks into coordination

A bare lock tells you something is busy. A claim with metadata tells you who owns it, where it is running, and what it is doing.

{
  "owner": "thumbnail-worker",
  "host": "server-b",
  "service": "media-pipeline",
  "task_id": "img-123",
  "job": "generate thumbnail",
  "claimed_at": "2026-04-01T03:55:00Z",
  "run_id": "run-8472",
  "run_url": "https://ops.example/runs/8472"
}

Fencing tokens make ownership safer

Every acquire returns a monotonically increasing fencing token. Downstream systems can reject stale writers even if a previous holder wakes up late.

UPDATE state SET payload = ?, fence = ? WHERE fence < ?;

This is how the lock becomes safer than best-effort mutual exclusion.
