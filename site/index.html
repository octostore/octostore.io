Self-hostable coordination over HTTP 🐙

Distributed locks and leases for local and multi-machine coordination

OctoStore is a self-hostable distributed lock and lease service over HTTP. Use it on one machine or across many machines to coordinate work with visible ownership and automatic recovery.

See coordination patterns → Read the docs
Primitive first: OctoStore is the lock and lease layer. On top of it, you can build simple coordination for scripts, workers, services, CI jobs, and agents.

Why it exists

When multiple workers can all see the same task, one of them should claim it and the others should back off.

⚠️

Duplicate work

Several workers, services, or agents all pick up the same task because they can all see the same backlog.

👀

Invisible ownership

Nobody can tell who owns a job, where it is running, or what it is doing right now.

♻️

Stuck work after crashes

A process dies and leaves the system looking busy forever unless ownership expires automatically.

Two natural ways to use it

The model stays the same. The deployment changes.

💻

Local coordination

Run OctoStore on one machine and let local scripts, cron jobs, workers, or agents coordinate through it.

🌐

Multi-machine coordination

Point workers on several servers at one shared OctoStore deployment and give them a shared ownership view.

🧩

General coordination patterns

Use the same claim, renew, release, and inspect pattern from Python, TypeScript, shell, CI, or any agent runtime.

What OctoStore gives you

A narrow primitive with real operational value.

🏆

One owner per job

Only one worker successfully claims a stable job name at a time.

📋

Visible ownership metadata

Attach host, service, run id, task id, and other details so anyone can inspect what is running where.

❤️

Heartbeat while running

Keep ownership alive while the worker is healthy and actively processing.

♻️

Automatic recovery

If the owner dies, the lease expires and another worker can safely take over.

🔢

Fencing tokens

Protect downstream writes from stale lock holders with monotonic tokens.

🌍

Simple HTTP API

Use it from any runtime without dragging in a heavyweight client or platform dependency.

The coordination pattern

Stable job name. Claim. Attach metadata. Renew while active. Release when done. Expire on failure.
  1. Choose one stable name for one logical unit of work.
  2. Have all workers try to claim the same name.
  3. One wins and publishes metadata.
  4. The others inspect the current owner and back off.
  5. The owner renews while healthy.
  6. If it dies, the lease expires and another worker can take over.
